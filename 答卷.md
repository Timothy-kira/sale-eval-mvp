# 第一部分：销售 Agent 故障诊断与重构

## 1. 诊断问题

| 对话 | 失败模式 | 具体问题 | 违反规则 |
|------|----------|----------|----------|
| **A** | 过度承诺 + 全幻觉 | ① 编造"完全替代销售、保证转化率提升50%"<br>② 编造价格"标准版5万/年，企业版20万/年"<br>③ 无任何工具调用<br>④ 未确认邮箱/时区/目的就口头"预约好了"<br>⑤ 无 CRM 记录 | 规则1（不得编造）<br>规则3（预约前确认三项信息）<br>规则5/8（CRM 记录） |
| **B** | 过早放弃 | ① 客户已提供公司规模、行业、痛点，仅因"预算未定"就结束对话<br>② 未追问决策人、上线时间等关键信息<br>③ 无 CRM 记录 | 规则4（澄清优先，不直接放弃）<br>规则8（主动更新 CRM） |
| **C** | 流程跳跃 + 虚假确认 | ① `check_calendar` 时区参数错误（新加坡→上海）<br>② 未确认参会目的<br>③ 未调用 `book_demo` 就声称"已约好"<br>④ `book_demo` 未成功就写 CRM"已预约"<br>⑤ CRM summary 未包含痛点<br>⑥ **用户首句即直接提供邮箱并索要 Demo，异常直白，存在竞争试探或信息收集风险，Agent 未识别异常也未转人工** | 规则3（确认参会目的）<br>规则5（CRM 内容完整）<br>规则6（预约成功才能写 CRM）<br>规则2/7（风险控制/安全） |

## 2. 改进方案

### 2.1 系统 Prompt

Prompt 文件路径：
- **v1**：`sales_agent/prompts/v1.md`
- **v2**：`sales_agent/prompts/v2.md`

**设计思路**：Prompt 与代码解耦，纯 Markdown + 模板变量注入，支持版本切换和 A/B 对比。

**核心规则**：

| 规则 | 内容 | 修复的问题 |
|------|------|------------|
| 1 | 不得编造价格/案例/功能；知识库有定价可引用，定制报价须确认 | 对话 A 编造价格/功能 |
| 2 | 合同/法务/安全/定制报价 → 转人工 | 风险控制 |
| 3 | Demo 前必须确认邮箱、时区、参会目的；缺一则追问 | 对话 A/C 预约违规 |
| 4 | 泛泛咨询时先澄清需求，不直接推销 | 对话 B 过早结束 |
| 5 | CRM summary 必须含：痛点、阶段、下一步、信心等级 | 对话 C CRM 内容缺失 |
| 6 | 不得把"已预约"写入 CRM，除非 `book_demo` 成功 | 对话 C 虚假 CRM |
| 7 | 安全：不得查询/修改其他 lead_id | 身份伪造防护 |
| 8 | 收集足够信息后 **主动** 写 CRM，不等用户要求 | 对话 B CRM 遗漏 |

### 2.2 工具调用策略

采用原生 OpenAI Function Calling：
- LLM 生成 `tool_calls` → 执行工具 → 结果以 `role="tool"` 回传 → 继续循环
- 支持多步工具链自动执行（如 `search_knowledge_base` → 根据结果回复）
- 单次可返回多个 tool_calls

### 2.3 转人工策略

| 层级 | 触发条件 | 响应 |
|------|----------|------|
| 硬编码层 | 消息包含其他 lead_id（L002/L003）妄图hack的 | 拦截，拒绝+转人工 |
| Prompt 层 | 合同/法务/安全/定制报价 | `handoff_to_human` |
| Prompt 层 | 修改 lead_id 或查询其他客户 | `handoff_to_human` |

### 2.4 防编造机制

三层防线：
1. **知识库优先**（COT 强制先调用 `search_knowledge_base`）→ 修复对话 A 的功能/价格编造
2. **负面示例**（7 个反例覆盖常见幻觉场景）→ 通过反例教学降低幻觉
3. **LLM 独立评判**（评测阶段语义评分，区分"事实引用"和"编造承诺"）→ 防止误判

## 4. Prompt 迭代（v1 → v2）

| 改动项 | v1 | v2 | 原因 |
|--------|----|----|------|
| 规则数 | 6 条 | 8 条 | 新增安全规则 + 主动 CRM 规则 |
| 思考流程 | 无 | 9 步 COT 模拟| 强制按流程思考，降低遗漏 |
| 负面示例 | 无 | 7 个 | 反例教学，强化错误记忆 |
| 知识库 | "不得编造" | "先检索，不得凭记忆回答" | 强化工具调用意识 |
| CRM 规则 | "每次交互后更新" | "信息足够后主动更新" | 更明确的触发条件 |

**关键改动说明**：

- **COT 思考流程**：v1 模型常"凭直觉回复"，跳过知识库检索或忘记 CRM。v2 增加 9 步固定流程（查看档案→理解意图→风险评估→信息缺口→知识检索→线索评级→下一步动作→工具决策→组织语言→输出 JSON），强制模型每步确认。效果：E003（知识库检索）从 FAIL 变 PASS。

- **负面示例**：v1 模型对"什么不能做"缺乏感性认识。v2 增加 7 个反例（编造案例、直接预约、价格编造、合同解释、CRM 虚假预约、越权查询、身份修改）。效果：E001（不编造价格）、E017（无预算不结束）从 FAIL 变 PASS。

**评测验证**：

| 指标 | v1 | v2 |
|------|----|----|
| 通过率 | 13/18 (72.2%) | 16/18 (88.9%) |
| v1 FAIL → v2 PASS | — | E001, E003, E017 |
| 共同失败 | — | E009（信息完整后未写 CRM）、E010（缺少 `check_calendar`） |

**遗留问题**：E009/E010 属于 COT 中"工具决策"步骤的流程刚性不足，需在后续迭代中固化为强制分支。
