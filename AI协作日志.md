## 2. 问题拆解

### 我最初如何理解问题

笔试分为三部分，核心是一个 B2B 销售 Agent 的开发与评测：
- **第一部分**偏产品设计：诊断现有 Agent 的问题，设计改进方案、评测集、Prompt 迭代
- **第二部分**偏工程实现：写出可运行、可评测的代码
- **第三部分**偏过程记录：证明我能高质量地使用 AI 工具

我的策略是**先搭一个最简陋的 Agent 框架，跑一轮评测分析 bad case，再针对性改进**——即：
1. 快速搭出能跑通的 v1（基础规则 Prompt + 硬编码评测）
2. 跑一轮评测，记录所有失败的 bad case
3. 分析 bad case 的根因，设计 v2（COT + 负面示例 + LLM 评判）
4. 再跑一轮对比，验证改进效果

这样第一部分的"Prompt 迭代"和"评测集设计"都有真实的数据支撑，不是凭空设计。

### 我把问题拆成了哪些子任务

| 阶段 | 子任务 | 优先级 |
|------|--------|--------|
| **Phase 1：最小可用 Agent** | ① 实现工具注册表 + Mock 工具（6 个）<br>② 实现 Agent 核心循环（Function Calling）<br>③ 实现输入/输出格式（JSON） | P0 |
| **Phase 2：评测体系** | ① 设计 EvalCase 数据结构<br>② 实现 Trajectory 硬编码检查<br>③ 实现 LLM 独立评判（judge 模式）<br>④ 实现 Mock DB 快照重置 | P0 |
| **Phase 3：Prompt 迭代** | ① 设计 v1 Prompt（基础规则）<br>② 设计 v2 Prompt（COT + 负面示例）<br>③ 跑对比评测，收集数据 | P0 |
| **Phase 4：文档与交付** | ① README（安装、运行、架构、已知问题）<br>② 第一部分答卷<br>③ 协作日志 | P1 |

### 哪些部分交给 AI，哪些部分必须自己判断

**交给 AI 的部分**：代码框架生成（Agent 循环、评测 runner、checker 骨架）、Prompt 初稿撰写（v1 规则清单、v2 COT 步骤）、评测用例初稿（从笔试题材料中抽取）、工具函数实现（Mock DB 的 JSON 读写）、文档结构建议和初稿。这些属于"生成型"任务，AI 可以快速产出可用草稿，节省大量时间。

**必须自己判断的部分**：badcase 的根因分析（从工具调用轨迹、回复语义中定位失败原因）、必须严格遵循规则的审核撰写（如"不得把已预约写入 CRM，除非 book_demo 成功"这类刚性规则的措辞和边界）、评测用例覆盖场景的审核和修改（判断 18 个 case 是否覆盖了所有关键场景、通过/失败标准是否可操作）、方案规划中的架构技术选择（如改用原生 Function Calling、拒绝插入全部 lead 档案改用当前 lead 隔离）。这些涉及业务理解、质量把关和技术决策，AI 无法替代。

**关键原则**：AI 擅长"生成"和"扩展"，但不擅长"判断什么是对的"。业务规则、评测标准、根因分析必须由自己把关，AI 的一次性输出mmvp只能作为草稿。

## 3. 关键 Prompt 记录

### Prompt 1

**Prompt**：
> "帮我注册工具，需要支持 6 个工具：get_lead_context、search_knowledge_base、check_calendar、book_demo、write_crm_note、handoff_to_human。输入是 lead_id + conversation，输出是 JSON（assistant_message、tool_calls、state）。"

**目的**：搭建 Agent 的核心骨架，让 Agent 能跑起来。

**AI 输出摘要**：
生成了 `sales_agent/agent/core.py` 的初版，包含：
- `agent_loop()` 主循环
- 工具注册表 `all_tools`


**我采纳了什么，没有采纳什么，原因**：
- **采纳**：整体架构（循环 + 工具注册）非常清晰，直接作为项目基础。
- **未采纳**：笔试题给定的输出格式要求模型在 `assistant_message` 中自行输出 JSON（含 `tool_calls` 字段），我选择**不采用该方案**，而是使用模型已支持的**原生 Function Calling**（`tools` + `tool_choice="auto"`）。原因：
  1. 让模型自行构造 JSON 格式不稳定，容易非法 JSON、字段缺失，导致反复解析-纠错-timeout
  2. 原生工具调用是模型在预训练阶段就已学习的范式，输出更可靠
  3. 工具结果以 `role="tool"` 回传，符合 OpenAI 标准，多轮工具链更顺畅

---

### Prompt 2

**Prompt**：
> "帮我做 v1 和 v2 prompt 的 A/B 对比评测框架。"

**目的**：建立可量化的 Prompt 迭代验证机制，用数据证明 v2 是否优于 v1。

**AI 输出摘要**：
AI 最初建议用硬编码关键词匹配做评测：
- `forbidden_in_message`：检查回复中是否包含禁止词（如"保证""一定"）
- `expected_in_message`：检查回复中是否必须包含某些词
- 加上 Trajectory 工具序列检查

**我采纳了什么，没有采纳什么，原因**：
- **未采纳**：拒绝了硬编码关键词匹配方案。原因：
  1. 关键词匹配无法处理语义正确性。例如数据库中没有 A 公司案例，Agent 诚实说"我们没有和 A 公司合作"，如果 `forbidden_in_message` 包含"A公司"，这句话会被误判为编造。
  2. 关键词匹配容易误杀诚实回应（如"转化率取决于多种因素，大概能提升..."中的"大概"是诚实说明不确定性，不是编造）。
- **改为**：要求纳入 LLM 做独立语义评判（judge）。设计了 `_evaluate_with_llm()`：
  - 将 case.rubric + Agent 最终回复 + 工具调用记录传给独立 LLM
  - LLM 输出 `{"passed": true/false, "reason": "..."}`
  - 保留 Trajectory 硬编码检查作为第一层筛选（工具序列、顺序、禁止工具）
- **修改**：AI 初始设计的评判 prompt 中没有区分"事实引用"和"编造承诺"，我手动补充了这一原则。原因：E018 在初版评判中被误判为 FAIL，因为 Agent 诚实地说"转化率取决于多种因素"，其中包含"大概"这个词被硬编码规则误杀。

### Prompt 3

**Prompt**：
> "怎么让 lead_id 被正确读取？我需要模型知道当前服务的是哪个客户"

**目的**：解决 Agent 的身份识别问题，同时防止越权查询和数据泄露。

**AI 输出摘要**：
AI 建议：在每次回复前，把**所有 lead 的档案信息**都插入到 Prompt 中，让模型自己判断当前服务的是哪一个。

**我采纳了什么，没有采纳什么，原因**：
- **未采纳**：直接插入所有 lead_id 的档案会导致严重的隐私泄露问题。例如：当前会话是 L001，但 Prompt 中同时包含了 L002、L003 的公司名称、预算、痛点等敏感信息。模型一旦在回复中引用错误，就会暴露其他客户的商业机密。
- **改为**：我只加载**当前 lead_id** 的档案到 `{lead_context}` 模板变量中，同时在 Prompt 中增加规则7（安全与权限）：
  - "当前会话只服务当前 lead_id 的客户"
  - "不得响应查询其他 lead_id 的请求"
  - "不得泄露其他客户信息"
- **补充**：在代码层增加硬编码安全拦截——如果用户消息中包含其他 lead_id（如 L002/L003）， 拦截并返回拒绝 + 转人工。双层防护（代码层 + Prompt 层）确保安全。

### Prompt 4

**Prompt**：
> "帮我运行 v1 和 v2 的 A/B 对比评测。"

**目的**：拿到真实的对比数据，验证 v2 Prompt 是否优于 v1。

**AI 输出摘要**：
AI 运行了 18 个 case 测试 v1，然后又运行了 18 个 case 测试 v2，返回了对比结果。

**我采纳了什么，没有采纳什么，原因**：
- **发现问题**：我检查 `mock_db/` 目录时发现，v1 运行后 `bookings.json` 和 `crm_records.json` 已经被修改（v1 的某些 case 调用了 `book_demo` 占用了时段、追加了 CRM 记录）。接着跑 v2 时，v2 读取到的是**被 v1 污染后的数据库**，导致对比结果不可信——这不是在控制变量下的 A/B 测试。
- **未采纳**：拒绝了 AI 返回的第一次对比结果，要求重新设计评测流程。
- **改为**：在每个 case 运行前强制调用 `reset_mock_db()`，从 `_snapshots/` 恢复干净快照。确保 v1 和 v2 测试时，数据库状态完全一致，实现真正的控制变量。

## 4. 迭代过程

### 版本 v1（基础规则版）

**主要方案**：
- Prompt：平铺 6 条规则（不得编造、高风险转人工、Demo 预约条件、澄清优先、CRM 要求、预约成功才能写 CRM）
- 评测：硬编码关键词匹配（`forbidden_in_message`、`expected_in_message`）+ Trajectory 工具序列检查
- Agent 循环：**初始版本让模型在 `assistant_message` 中直接输出 JSON 格式的 `tool_calls`**（即笔试给定的输出格式），而非原生 Function Calling。测试发现该方式频繁 timeout——模型需要自行构造 JSON 格式，容易出错、需要多轮纠错。后改为原生 `tools` + `tool_choice="auto"` 接口。

**发现的问题**：
1. **E001 FAIL**：Agent 编造了价格区间（"大概 5-10 万/年"）。根因：规则1"不得编造"太笼统，模型没有"先查知识库"的肌肉记忆。
2. **E003 FAIL**：用户问产品功能，Agent 凭记忆回答，未调用 `search_knowledge_base`。根因：Prompt 没有强制思考"我是否知道这个事实"。
3. **E017 FAIL**：用户说"预算还没定"，Agent 直接结束对话。根因：模型把"无预算"等同于"无意向"，缺乏"继续挖掘需求"的指令。
4. **E018 误判**：Agent 诚实回应"转化率取决于多种因素"，但硬编码规则因包含"大概"一词判为 FAIL。根因：关键词匹配无法区分"编造承诺"和"诚实说明不确定性"。


### 版本 v2（COT + 负面示例版）

**改动**：
1. **Prompt 文件真正加载**：`agent_loop()` 从 `sales_agent/prompts/{version}.md` 动态读取，模板变量注入上下文。
2. **增加 9 步 COT 思考流程**：强制模型按"查看档案→理解意图→风险评估→信息缺口→知识检索→线索评级→下一步动作→工具决策→组织语言→输出 JSON"的顺序思考。
3. **增加 7 个负面示例**：覆盖编造案例、直接预约、价格编造、合同解释、CRM 虚假预约、越权查询、身份修改。
4. **增加 2 条规则**：规则7（安全：禁止查询/修改其他 lead_id）、规则8（主动更新 CRM）。
5. **评测改为 LLM 独立评判**：移除硬编码关键词匹配，改用 rubric + LongCat-2.0 语义评判。
6. **Mock DB 快照重置**：每个 case 前自动恢复干净状态，避免 case 间污染。

**效果**：
| 指标 | v1 | v2 |
|------|----|----|
| 通过率 | 13/18 (72.2%) | 16/18 (88.9%) |
| 差异 case（v1 FAIL → v2 PASS） | — | E001（不编造价格）、E003（检索知识库）、E017（无预算不结束） |
| 共同失败 | — | E009（信息完整后未写 CRM）、E010（缺少 check_calendar） |

## 5. 评测或验证过程

### 验证方式

1. **A/B 对比评测**：每个 case 分别用 v1 和 v2 prompt 运行，记录完整 trajectory 和最终输出。
2. **分层评分**：
   - 第一层：Trajectory 硬编码检查（工具序列、顺序、禁止工具）
   - 第二层：LLM 独立评判（Message Content 语义正确性）
3. **分 case 日志**：每个 case 生成独立 JSON 文件，包含 v1/v2 的全量对比数据，便于逐 case 复盘。

### 失败的 case 及修复

| case | 问题 | 修复方式 |
|------|------|----------|
| **E018 误判** | 硬编码规则因"大概"一词判 FAIL | 改为 LLM 语义评判，评判 prompt 中增加"事实引用≠编造承诺"原则 |

| **E010 共同失败** | v2 顺序为 `book_demo → write_crm_note`，缺少 `check_calendar` | **未修复**，根因为 COT 未固定 Demo 预约 SOP，记录为已知问题 |

## 6. AI 输出错误记录

### 错误 1：建议用硬编码关键词匹配做评测

**AI 建议**：
> "评测可以用 `forbidden_in_message` 和 `expected_in_message` 列表，检查 Agent 回复中是否包含特定关键词。"

**为什么不对**：
关键词匹配无法处理语义层面的正确性。例如：
- 数据库中没有 A 公司的合作案例，Agent 诚实回复"**我们没有和 A 公司合作**" → 如果硬编码规则设置了 `forbidden_in_message: ["A公司"]`（禁止提到 A 公司），这句话会被误判为 FAIL（编造案例），尽管它完全是诚实的否定回答。
- Agent 说"**我们没有这个功能**" → 包含 "没有"，但这是**正确**回应（承认 limitations）。
- Agent 说"**大概能提升 30%**" → 包含 "大概"，但如果是**引用知识库中的真实案例数据**，这是事实引用而非编造。

**我是如何发现并修改的**：
在 E018 评测中，Agent 诚实回应"转化率取决于多种因素，无法承诺具体数字"，但硬编码规则因检测到"大概"一词判为 FAIL。我意识到这种评判方式不可持续，改为让独立 LLM 根据 rubric 做语义评判，并在评判 prompt 中明确"事实引用≠编造承诺"。

---

### 错误 2：生成的 `agent_loop()` 硬编码了 prompt 字符串

**AI 建议**：
> "`agent_loop()` 中可以直接定义 system prompt 字符串，这样简单直接。"

**为什么不对**：
如果 prompt 硬编码在 Python 文件中，会导致：
1. 无法做 A/B 对比评测（v1 vs v2 切换需要改代码重新部署）
2. 产品/运营无法独立修改 prompt
3. 代码与业务逻辑耦合，违背"配置与代码分离"原则

**我是如何发现并修改的**：
在第一次运行 v2 Prompt 后，发现结果与 v1 完全一致。检查日志发现 `agent_loop()` 中根本没有读取 `sales_agent/prompts/v2.md` 文件，而是使用了代码中硬编码的字符串。我要求 AI 重写为从文件动态加载，并通过模板变量注入 `lead_context`、`missing_info` 等动态上下文。

---

### 错误 3：建议模型在 assistant_message 中自行输出 JSON 格式的 tool_calls

**AI 建议**：
> "Agent 直接在 `assistant_message` 中输出 JSON，包含 `tool_calls` 字段，然后由程序解析执行。"

**为什么不对**：
让模型自行构造 JSON 格式的工具调用存在严重问题：
1. **格式不稳定**：模型经常输出非法 JSON（缺少引号、括号不匹配、字段名错误），需要反复解析-纠错-重试
2. **容易 timeout**：一轮对话中可能因格式错误重试 3-4 次，导致总耗时超过 timeout 阈值
3. **无法利用模型内置的工具调用能力**：现代 LLM（如 GPT-4、LongCat-2.0）支持原生 Function Calling，模型在底层就已训练过工具调用范式，输出更稳定

**我是如何发现并修改的**：
接入真实 LongCat-2.0 API 测试时，发现大量 case 因"模型输出格式错误"而 timeout 失败。检查日志发现模型在 `assistant_message` 中输出的 JSON 经常缺少字段或格式错误。我要求将 Agent 循环改为原生 Function Calling（`tools` 参数 + `tool_choice="auto"`），让模型通过标准接口返回 `tool_calls`，工具结果以 `role="tool"` 回传。改造后 timeout 问题基本消失，工具调用成功率大幅提升。

---

### 错误 4：A/B 对比评测未隔离数据库状态

**AI 建议**：
> "直接在一个循环里先跑 v1 再跑 v2，对比结果即可。"

**为什么不对**：
先跑的 prompt 版本会修改 Mock DB（如 `book_demo` 占用时段、`write_crm_note` 追加记录），导致后跑的版本读取到的数据库状态是脏的。例如：
- v1 先跑，某个 case 调用了 `book_demo(slot_1)`，把 slot_1 标记为已占用
- v2 后跑同一个 case，`check_calendar` 返回的可用时段里已经没有 slot_1 了
- 这不仅影响当前 case，还可能影响后续 case 的评测结果，A/B 对比完全不公平

**我是如何发现并修改的**：
在第一次跑对比评测时，发现 v2 的某些 case  behavior 异常（如返回的时段与 v1 不同）。检查 `mock_db/` 下的 JSON 文件，发现 bookings.json 和 crm_records.json 在 v1 运行后已经被修改。我要求在每个 case 运行前调用 `reset_mock_db()`，从 `_snapshots/` 恢复干净快照。改造后 case 间完全隔离，A/B 对比结果可信。

---

### 错误 5：没有建立工具依赖关系图 / Workflow SOP

**AI 建议**：
> "用负面示例告诉模型什么不能做，比如'不能直接调用 book_demo'。"

**为什么不对**：
仅靠负面示例和 COT 文字描述，无法让模型形成清晰的"工具调用依赖关系"。模型需要一张明确的 Workflow 图或依赖关系表，告诉它：
- 预约 Demo 的正确路径：`确认信息 → check_calendar → book_demo → write_crm_note`
- 更新 CRM 的触发条件：`missing_info 为空 → write_crm_note`
- 转人工的决策树：`合同/法务/安全/定制报价 → handoff_to_human`

没有这张图，模型在 E010 上知道"不能直接预约"，但仍然跳过了 `check_calendar`；在 E009 上知道"要主动写 CRM"，但执行时只做了 `update_state` 没做 `write_crm_note`。

**我是如何发现并修改的**：
在评测 E010 时发现 v2 的工具序列为 `book_demo → write_crm_note`，缺少 `check_calendar`；E009 只做了 `get_lead_context → update_state`。根因不是 Prompt 文字不够多，而是缺少一张显式的"工具依赖关系图"。后续迭代应在 Prompt 中加入 Mermaid 图或结构化表格，明确每个场景的 Step-by-Step SOP。

## 7. 最终反思

1. **没有重复实验**：当前评测为单次运行，LLM 输出存在方差，单次结果可能无法反映 prompt 的真实上限。应增加 `n=3` 重复运行，统计通过率均值与方差，识别 flaky case。

2. **Prompt 过于简陋，应做分层策略**：当前 v2 是一个单一文件，所有规则、混在一起。应拆分为分层结构：
   - **System Prompt**：角色定义 + 核心规则（不变层）
   - **Dynamic Context**：当前线索档案 + 状态（每次注入）
   - **SOP Cards**：Demo 预约、CRM 记录、转人工等场景的标准操作卡片（按需加载）
   - **Few-shot 示例库**：按场景分类的正面/反面示例
   这样可以精准控制每个场景的信息量，避免模型在长 prompt 中遗漏关键规则。

3. **模拟用户应交给 LLM 测试**：当前测试用例的用户输入是人工编写的静态文本（`static_cases.json`），对话轮次和意图变化有限。应引入 LLM-based User Simulator：
   - 让 LLM 扮演真实客户，根据场景目标（如"试探价格""钓鱼查询""真正想预约"）动态生成回复
   - 可以覆盖人工难以想到的边缘表达和对抗性输入
   - 与 Agent 形成多轮动态博弈，更贴近真实上线环境

### 如果这个 Agent 要上线，我最担心什么

1. **查询数据库太粗糙，应考虑 RAG**：当前 `search_knowledge_base` 是简单的 JSON 文件关键词匹配，召回率和准确率都很低。知识库一旦增大，模型更容易"查不到就瞎编"。应引入向量检索（RAG）+ 重排序，让知识库查询更精准，降低幻觉概率。

2. **安全测试不足，存在被劫持风险**：当前仅测试了 E012/E013（直接查询其他 lead_id），但对抗性攻击远不止于此。
   hack prompt 在当前评测集中完全未覆盖，上线后一旦被攻破，会直接泄露内部资料和其他用户画像。
